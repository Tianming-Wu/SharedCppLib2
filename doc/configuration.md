# Configuration Documentation

This document provides an overview of the configuration system in SharedCppLib2.

To be clear, SharedCppLib2 does not have a built-in configuration system, but it provides a set of tools and interfaces to help you build your own configuration system.

Below are the options you can choose to use.

If you cannot select one, use ini for simple purposes, json for structured data, and keydb for heavy binary data.



# Getting Started

### Json Format

SharedCppLib2 now provides a built-in JSON module. It is still in early development — its functionality may not yet match feature-rich libraries like [jsoncpp](https://github.com/open-source-parsers/jsoncpp), but it is lightweight and can actually be used independently of SharedCppLib2 (just copy the header and source file).

Some of its syntax is intentionally designed to be similar to jsoncpp for familiarity, but mostly it follows what I think is convenient.

```cpp
#include <SharedCppLib2/json.hpp>

int main() {
    // Parse from string
    scl2::json j = scl2::json::fromString(R"({
        "name": "example",
        "version": 1,
        "features": ["a", "b"],
        "enabled": true
    })");

    // Access values
    std::string name = j["name"].as_string();
    int64_t version = j["version"].as_int();
    bool enabled = j["enabled"].as_bool();

    // Array access
    for (const auto& elem : j["features"].as_array()) {
        std::cout << elem.as_string() << std::endl;
    }

    // Build and export
    scl2::json output = scl2::json::fromString(R"({"key": "value"})");
    output["new_field"] = scl2::json_value(42);
    std::string text = output.toString();   // formatted output
    std::string compact = output.toCompatString();  // compact output

    // File I/O
    scl2::json cfg = scl2::json::fromFile("config.json");
    cfg.toFile("config_updated.json");

    return 0;
}
```

> [!WARNING]
> The JSON module is in early development. While usable, it may lack some advanced features found in dedicated JSON libraries. Expect improvements in future releases.

### Ini Format

You can easily manage configuration with the [`ini`](./ini.md) module.

```cpp
#include <SharedCppLib2/ini.hpp>

int main() {
    ini config;

    if(!config.loadFromFile("config.ini")) {
        std::cerr << "Failed to load config file." << std::endl;
        return 1;
    }

    std::string name = config.getValue("main", "name", "default_name");
    config.setValue("main", "version", "1.0.0");

    int i = config.getValueAsInteger<int>("main", "max_connections", 100);

    // setValue supports any value that can be converted to string using std::to_string, and also supports string directly.

    // Boolean values are supported.
    bool debug_mode = config.getValueAsBool("main", "debug_mode", false);
    config.setValue("main", "debug_mode", !debug_mode);

    // Ini library also supports SharedCppLib2 API intergration. We won't go into details here, check ini documentation for more info.


    // You can use optional series functions to check if the value exists. It makes the code more complex though.
    auto w = config.getValueAsIntegerOptional<int>("main", "window_width");
    if(w.has_value()) {
        std::cout << "Window width: " << w.value() << std::endl;
    } else {
        std::cout << "Window width not found, using default value." << std::endl;
    }

    config.saveToFile("config.ini");
    return 0;
}
```

### Bytearray Configuration

SharedCppLib2 designed a standard procedure to store configuration.

This is meant to make the whole process clean, controllable and efficient. All the works are handled by the module itself, and leave the top-level code as simple as 1-2 lines (without compression).

It allows you to transform any kind of data into a byte array. The generated byte array can then be stored in a file, sent over the network, or get encrpyted, etc.


The below examples uses struct to define the config. class works as well, and they are basically the same in C++.
The only difference is that for a class, you MUST make the load and dump functions public.

If you need to make a constructer for the struct, make sure to leave an empty constructer as well (it can be private if you want), since your load probably needs to construct an empty struct first. However, if you can write your load function not using that, feel free to not do so.

#### Definition of plain config structure

Plain config structure is a struct that only contains trivial data types, meaning that they can be directly copied without any special handling. For example:
```cpp
struct MyConfig {
    int window_width;
    int window_height;
    bool debug_mode;
};
// This struct is considered plain because it only contains trivial data types (int and bool).
```
Plain structures are directly supported, however you need to construct the bytearray manually for now since this part of the API is still in development. It is still simple though.

```cpp
myConfig cfg{800, 600, true};
std::bytearray ba(cfg); // as simple.

myConfig cfg2 = ba.as<MyConfig>(); // and also simple.
```

If it is meant to be embedded, you can also make the load and dump functions for the struct. The only difference is the load function, where you need to specify the size:
```cpp
myStruct myStruct::load(const std::bytearray_view& data) {
    return data.readBytes(sizeof(myStruct)).as<myStruct>();
}
```
> Also, this step is because of the incomplete API. The final API will simplify this process, making you do not need to write a single line to handle this. 

#### Definition of non-plain config structure

For example, if you have a struct that you can use as configuration:
```cpp
struct MyConfig {
    std::string name;
    bool debug_mode;
    LogLevel log_level;
};
```

For the API to work, it's suggested to use the following method:
```cpp
struct MyConfig {
    // ... previous definition ...

    static MyConfig load(const std::bytearray_view& data);
    static std::bytearray dump(const MyConfig& config);
};

// You can use this macro to do an assert about whether you have written the API correctly:
scl2_check_generic_dump_load(MyConfig);
```
Notice that bytearray_view is used here instead of bytearray, since bytearray_view handles the cursor for you, and is the key part to make the code clean and easy to read.

Then you need to implement the load and dump functions.

The ONLY thing you need to ensure is that these functions are consistent with each other. That is, if you dump a config and then load it, you should always get the same config back. You have to make sure of that by yourself, since by no means can I handle this for you.

```cpp

MyConfig MyConfig::load(const std::bytearray_view& data) {
    MyConfig config;

    // For string, use the provided readString.
    // You should use readString() and addString() in pairs.
    config.name = data.readString();

    // For wide strings, there are also readWString and addWString.
    // All other string types like u8string and u16string are not
    // supported by the view API, but they are supported by the
    // bytearray API. You can handle them manually if you want. 

    // For most of the plain types, use the template read.
    config.debug_mode = data.read<bool>();
    config.log_level = data.read<LogLevel>();

    return config;
}

std::bytearray MyConfig::dump(const MyConfig& config) {
    std::bytearray ba;

    // You can choose to pre-allocate the bytearray. The grammar is the same as STL containers.
    ba.reserve(256); // This is not necessary, but it can improve performance by reducing the number of memory allocations.
    
    // For string, use the provided addString. This function handles the length of the string for you.
    ba.addString(config.name);

    // For most of the data types, especially numeric types (enum included), you can directly append them to the bytearray.
    ba.append(config.debug_mode);
    ba.append(config.log_level);

    return ba;
}
```

#### Definition of embedded config structure
Embedded config structure is a struct that contains other config structures. For example:
```cpp
struct AppConfig {
    MyConfig myConfig;
    scl2::logmgr_config logmgrConfig;

    // define the load and dump the same way here.
};
```

And here's the amazing part. Load/dump functions can be called recursively for sub-structures.
```cpp
AppConfig AppConfig::load(const std::bytearray_view& data) {
    AppConfig config;

    // Just call the load function of the sub-structure, and it will handle everything for you.
    // Still need to do it in a constant sequence though.
    config.myConfig = MyConfig::load(data);
    config.logmgrConfig = scl2::logmgr_config::load(data);

    return config;
}

std::bytearray AppConfig::dump(const AppConfig& config) {
    std::bytearray ba;

    // Just call the dump function of the sub-structure, and it will handle everything for you.
    ba.append(MyConfig::dump(config.myConfig));
    ba.append(scl2::logmgr_config::dump(config.logmgrConfig));

    return ba;
}
```


#### Bytearray handling
Here's how your top-level code should interact with the above API.

##### Generic
Except some designed Single-Line APIs to simplify the process, here is the generic method to handle your configuration. Using the traditional file method as an example:

```cpp
#include <SharedCppLib2/api.hpp>
#include <fstream>

int main() {
    {
        // You must open the file in binary mode for any Read api work correctly.
        std::ifstream cfgf("config.db", std::ios::binary);

        std::bytearray ba;
        cfgf >> ba;

        // It's not pretty clear here, but NO MATTER WHAT, you must let the current scope own
        // the loaded bytearray and make sure it is within its life cycle until you finish
        // loading the config.
        // This is because the bytearray_view does not own the data, and it is your responsibility
        // to ensure that the data is valid until you finish loading the config.
        MyConfig config = MyConfig::load(ba);

        // If you are not happy with the temporary bytearray object, you can clear it after using it.
        ba.clear();
    }

    {
        std::bytearray outba = MyConfig::dump(config);
        std::ofstream outcfgf("config.db", std::ios::binary);
        outcfgf << outba;
    }
}
```

##### File
You can use the SharedCppLib2 fileio Single-Line API to handle the file reading and writing.

```cpp
#include <SharedCppLib2/fileio.hpp>

// It's suggested to use the "db" suffix because it looks cool (

MyConfig config = scl2::readAndLoad<MyConfig>("config.db"); // This requires "load" support.

scl2::writeFile("config.db", config); // This requires "dump" support.

// It's suggested to use an absolute path for the file, since you probably don't want you config file to be in the current working directory. 
// You can use the platform API like this:

#include <SharedCppLib2/platform.hpp>
std::filesystem::path configPath = platform::executable_dir() / "config.db";
```

##### Encryption
SharedCppLib2 also provides an encryption API definition set, and it is included in the SharedCppLib2 Generic API.

However, currently there is no encryption algorithm implemented in SharedCppLib2. For any satisfied classes that is written by users (see Encryption API documentation for more details), it should be simple:

```cpp
#include <SharedCppLib2/api.hpp>
#include <AES256.hpp> // This is an example placeholder for a supported encryption algorithm.

std::bytearray cfgData = MyConfig::dump(config);

std::bytearray enc = scl2::encrypt<AES256>(cfgData, key);
scl2::writeFile("config.enc", enc);

std::bytearray uecfgData = scl2::decrypt<AES256>(enc, key);
```
